# 1C Email Contact Field Mapping

Дата обновления: 2026-05-10

Назначение: описать поля, которые нужны для связки email-переписки с 1С. Главный принцип: переписка должна привязываться не только к лиду/контрагенту, а к конкретному контакту внутри лида/контрагента, потому что у одной компании может быть несколько людей и адресов.

1С на этом этапе является только read-only источником. Ничего в 1С не создаем и не меняем. Новые данные, результаты сопоставления, история переписки и AI-выводы пишутся в отдельную PostgreSQL базу `marketing_db`.

Пользователь 1С для ручных read-only тестов:

```text
Артем
```

## Стартовый сценарий

Сначала только чтение:

```text
email -> найти контакт по email/телефону -> определить владельца контакта -> определить лид или контрагента -> записать результат в dry-run таблицу/лог
```

Запись в 1С запрещена. Запись разрешена только в `marketing_db` после создания схемы.

## Правила пустых полей

Нельзя пропускать лида/контрагента только потому, что не заполнены:

```text
ИНН
теги
группа
заметки
адрес
сайт
контакты
телефон
часть контактной информации
```

Пропуск разрешен только если одновременно:

```text
нет имени/наименования
и
нет e-mail
```

Или если по совокупности признаков объект относится к спаму/мусору:

```text
теги + заметки/комментарий + группа/источник + прочие признаки
```

Пустой ИНН сам по себе не является признаком мусора.

## Email и контакты

Email может быть:

1. на уровне лида или контрагента;
2. на уровне каждого отдельного контакта;
3. одновременно в нескольких местах;
4. отсутствовать.

Контактов может быть несколько или не быть вообще. Если контакты есть, для `marketing_db` нужно собирать все доступные контактные данные по каждому контакту.

## Name and address fallback

Наименование:

```text
если есть Наименование в документах -> использовать его
иначе использовать Наименование / Description / НаименованиеКомпании
```

Адрес:

```text
если есть юр. адрес -> использовать его
иначе факт. адрес
иначе любой доступный адрес
иначе пусто
```

Отсутствие адреса не является причиной пропуска.

## Группа как бизнес-признак

Группа важна как признак первого интереса клиента. Примеры:

```text
ABB
Festo
Mitsubishi
покупка оборудования
поиск товара
импорт под ключ
доставка
проверка поставщика
другой бренд
```

Группа используется для классификации и приоритизации, но не является обязательным полем.

## Первый порядок поиска

1. Искать email на уровне лида: `Catalog_Лиды.АдресЭПДляПоиска`.
2. Искать email на уровне контрагента: `Catalog_Контрагенты.АдресЭПДляПоиска`.
3. Искать email в `Catalog_КонтактыЛидов_КонтактнаяИнформация`.
4. Если найдено, взять `Ref_Key` контакта и через `Catalog_КонтактыЛидов` получить `Owner_Key` лида.
5. Искать email в `Catalog_КонтактныеЛица_КонтактнаяИнформация`.
6. Искать email в `Catalog_Контрагенты_КонтактнаяИнформация`.
7. Если не найдено по email, повторить по телефону.
8. Если не найдено, создать dry-run запись `unknown contact` в `marketing_db` или временном логе; в 1С ничего не создавать.

## Важное ограничение

Не выполнять запись в 1С. Все действия с 1С только read-only. Результаты сопоставления и новые данные складывать в `marketing_db`.
