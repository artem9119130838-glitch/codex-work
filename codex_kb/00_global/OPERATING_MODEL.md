# Operating Model

## Цель базы знаний

Codex должен помнить не отдельный чат, а устойчивую систему вашей работы:

- какие постоянные объекты существуют;
- какие направления повторяются;
- какие решения уже приняты;
- какие процедуры уже проверены;
- какие риски нельзя забывать;
- где искать детали, если краткой базы мало.

## Уровни знания

### 1. Корень workspace

Файлы:

- `E:\Codex_Work\AGENTS.md`
- `E:\Codex_Work\CODEX_START_HERE.md`

Назначение: дать любому новому чату общий контекст и ссылку на базу знаний.

### 2. Общая база знаний

Папка:

```text
E:\Codex_Work\codex_kb
```

Назначение: хранить стабильные факты и повторяемые процедуры для всех проектов.

### 3. Корень конкретного проекта

В каждом проекте желательно иметь:

- `AGENTS.md` - что важно именно для этого проекта.
- `PROJECT_CONTEXT.md` - текущее состояние проекта.
- `RUNBOOK.md` - команды именно этого проекта.
- `TODO.md` - открытые задачи.

Шаблоны лежат в `codex_kb\templates`.

В новый чат отправляется минимальный стартовый контекст:
- Сводка сессии: `.ai/SESSION_SUMMARY.md` (или `PROJECT_HANDOFF.md`);
- Краткая архитектура: `.ai/architecture.md`;
- Индекс файлов: `.ai/file_index.json`.

Все остальные доменные и прогресс-файлы ИИ считывает прогрессивно, точечно и только при необходимости.


## Что считать неизменным

Неизменное не значит вечное. Это факты, которые меняются редко и должны переиспользоваться:

- серверы, домены, сервисы, пути;
- роли систем: 1С, n8n, Bitrix, почта, Google Calendar;
- локальный workspace;
- роутер/VPN-модель;
- телефон и голосовой ввод;
- правила безопасности;
- проверенные команды и типовые ошибки.

## Что не хранить в базе

- пароли;
- токены;
- приватные ключи;
- полные webhook URL с секретами;
- персональные данные клиентов без необходимости;
- сырые большие логи, если достаточно короткого вывода и ссылки на файл.
