# TOKEN-FIRST VIBE ENGINEERING
## Итоговый регламент промышленного вайб-кодинга с минимальным расходом токенов

Версия: 1.0  
Статус: Рекомендуемый стандарт  
Цель: максимальное снижение расхода API-токенов и стоимости работы ИИ без существенного ухудшения качества разработки.

---

# 1. Цели системы

Система должна обеспечивать:

- минимальный расход токенов;
- предсказуемые затраты;
- невозможность случайного перерасхода;
- сохранение качества инженерных решений;
- масштабируемость на большие проекты;
- автоматический контроль бюджета;
- возможность использования нескольких моделей одновременно.

---

# 2. Основной принцип

Самая большая экономия достигается не за счёт сокращения ответов модели.

Самая большая экономия достигается за счёт предотвращения передачи лишнего контекста.

Следовательно:

> Главный объект оптимизации — контекст.

---

# 3. Архитектура Project Memory Layer

## Почему изменено

Изначальные планы предполагали чтение файлов проекта по мере необходимости.

Даже при ограничениях это приводит к постоянному повторному чтению кода.

Основной источник перерасхода устраняется созданием постоянной памяти проекта.

---

## Структура

```text
.ai/
├── architecture.md
├── file_index.json
├── dependency_graph.json
├── decisions.md
├── summaries/
└── budgets.yaml
```

---

## architecture.md

Содержит краткое описание системы.

Объём:

```text
500–1000 токенов
```

Содержимое:

- назначение проекта;
- основные модули;
- архитектурные ограничения;
- запрещённые решения;
- используемые технологии.

Пример:

```markdown
Auth:
- JWT
- login.py
- refresh.py

Billing:
- stripe_adapter.py
- invoices.py

Запрещено:
- OAuth
- Session Storage
```

---

## file_index.json

### Изменение

Вместо чтения файлов ИИ сначала работает с индексом.

---

Пример:

```json
{
  "auth/login.py":
    "JWT validation and session creation",

  "crm/bitrix.py":
    "Bitrix API wrapper",

  "billing/invoice.py":
    "invoice generation"
}
```

---

Экономия:

```text
90–97%
```

относительно передачи полного проекта.

---

## decisions.md

Хранит только архитектурные решения.

Пример:

```markdown
2026-06-01

Выбрали RabbitMQ вместо Kafka.

Причина:
небольшой объём сообщений.
```

# 4. Progressive Context Loading

## Почему изменено

Предыдущие планы ограничивали чтение файлов.

Но они не определяли порядок расширения контекста.

Вводится обязательная лестница доступа.

---

## Уровень 1

Передаётся только summary.

Пример:

```text
login.py

JWT validation
Session creation
Password reset
```

Расход:

```text
20–100 токенов
```

---

## Уровень 2

Передаются конкретные строки.

Например:

```text
45–78 строки
```

Расход:

```text
100–500 токенов
```

---

## Уровень 3

Передаётся весь файл.

Только при необходимости.

---

## Уровень 4

Передаются связанные файлы.

Только после обоснования.

---

Запрещено:

```text
читать весь проект сразу.
```

---

Экономия:

```text
70–90%
```

---

# 5. Router моделей

## Почему изменено

Использование одной модели для всех задач создаёт огромный перерасход.

Модели должны использоваться по назначению.

---

## Маршрутизация

```text
Запрос
↓
Task Classifier
↓
Flash-8B / Local
↓
Flash
↓
Pro
```

---

## Flash-8B / Local

Используется для:

- grep;
- поиск;
- классификация;
- документация;
- summary;
- naming;
- генерация тестов;
- переписывание текста.

Ожидаемая нагрузка:

```text
60–70%
```

---

## Flash

Используется для:

- написания функций;
- исправления багов;
- небольшого рефакторинга;
- анализа до 5 файлов.

Ожидаемая нагрузка:

```text
25–35%
```

---

## Pro

Используется только для:

- архитектурных решений;
- сложной отладки;
- анализа более 5 файлов;
- проектирования.

Ожидаемая нагрузка:

```text
5–10%
```

---

Экономия:

```text
50–80%
```

---

# 6. Preflight Gateway

## Почему изменено

Ручной калькулятор токенов игнорируется.

Расчёт должен происходить автоматически.

---

Перед каждым вызовом API:

```text
Оценить токены
↓
Рассчитать стоимость
↓
Проверить лимиты
↓
Сравнить с бюджетом
↓
Разрешить или отклонить
```

---

## Конфигурация

```yaml
request_budget_usd: 0.01
session_budget_usd: 0.50

limits:

  flash:
    max_input: 20000
    max_output: 3000

  pro:
    max_input: 40000
    max_output: 4000
```

---

Если стоимость превышена:

```text
Запрос оценивается в $0.036.

После оптимизации:
$0.005.

Подтвердите выполнение.
```

---

# 7. Budget Enforcer

## Почему изменено

Отчёты о расходах не предотвращают перерасход.

Нужна система принудительных ограничений.

---

## Дневные бюджеты

```yaml
daily:

  pro: 5
  flash: 20

weekly:

  total: 50

monthly:

  total: 150
```

---

## Автоматический дегрейд

```text
Pro недоступен
↓
Flash
↓
Flash-8B
↓
Локальная модель
```

---

Если бюджет исчерпан:

```text
Сессия автоматически завершается.
```

---

# 8. Session Compression

## Почему изменено

Большие чаты становятся главным источником расходов.

История должна заменяться краткими итогами.

---

После завершения работы создаётся:

```text
SESSION_SUMMARY.md
```

---

Пример:

```markdown
Исправлено:
- JWT bug

Изменены:
- login.py
- auth_tests.py

Открытые вопросы:
- refresh token expiration
```

---

Размер:

```text
150–300 токенов
```

---

Экономия:

```text
80–95%
на длинных сессиях.
```

---

# 9. Жёсткие лимиты AGENTS.md

## Почему изменено

В предыдущих планах ограничения были рекомендательными.

Теперь они обязательны.

---

## TOKEN_LIMITS

| Действие | Лимит |
|----------|--------|
| Чтение файла | ≤ 200 строк |
| Полное чтение файла | Только после summary |
| grep | ≤ 20 файлов |
| Глубина поиска | ≤ 3 уровня |
| Автоисправления | ≤ 3 попыток |
| История субагента | ≤ 2000 токенов |
| Ответ модели | ≤ 4000 токенов |

---

Нарушение:

```text
Критическая ошибка.
Работа должна быть остановлена.
```

# 11. Vibe Coding Protocol

## Почему изменено

Человеческий фактор является одной из основных причин перерасхода.

---

Каждый запрос оформляется так:

```markdown
ЗАДАЧА:
...

ФАЙЛЫ:
...

ДИАПАЗОН:
...

РЕЗУЛЬТАТ:
...

НЕ ДЕЛАТЬ:
...
```

---

Пример:

```markdown
ЗАДАЧА:
Исправить validate_token.

ФАЙЛЫ:
auth/login.py

ДИАПАЗОН:
45–78

РЕЗУЛЬТАТ:
Unified diff.

НЕ ДЕЛАТЬ:
Не читать другие файлы.
Не выводить полный код файла.
```

---

Использовать пакетную обработку для:

# 14. Мониторинг расходов

Система должна вести журнал:

```text
token_usage.jsonl
```

---

Для каждого вызова сохраняется:

- модель;
- input;
- output;
- cache;
- стоимость;
- длительность;
- задача.

---

Отображать:

- топ-10 дорогих запросов;
- расход по моделям;
- расход по проектам;
- расход по пользователям.

---

# 15. Верификация

## Автоматические тесты

Проверить:

- расчёт стоимости;
- соблюдение лимитов;
- отказ при превышении бюджета;
- fail-fast после 3 попыток;
- соблюдение .aiignore;
- progressive loading.

---

## Стресс-тесты

Проверить:

- запрос на чтение 100 файлов;
- неисправимый баг;
- превышение бюджета;
- длинную сессию более 100 запросов.

# Итог

Основная идея данного стандарта:

> Не экономить токены после отправки запроса.

> Не допускать появления лишних токенов вообще.

Контекст становится индексируемым и прогрессивно раскрываемым, модели используются строго по назначению, бюджеты контролируются автоматически, а длинные диалоги заменяются краткими машинными сводками.

Это позволяет построить промышленную систему вайб-кодинга с предсказуемой стоимостью и снижением расходов на API до 90–98% без заметной потери качества разработки.
