# Шаблон плана реализации: Универсальная 3-уровневая система ИИ-правил и безопасность Git Push

Этот документ представляет собой готовый шаблон плана для развертывания единых стандартов работы с ИИ-ассистентами и ограничения внешних Git-пушей. Предназначен для копирования и внедрения сотрудниками в их рабочих окружениях.

---

## 1. Концепция 3-уровневой системы правил ИИ

Для максимальной экономии токенов и предсказуемости поведения ИИ-агентов вся система правил делится на три уровня:

```
Рабочее пространство проекта
├── MANIFEST.md              ← 1. Сверхлегкие метаданные (до 15 строк)
├── AGENTS.md                ← 2. Точка входа / ИИ-контракт (до 2 страниц)
├── ARCHITECTURE.md          ← 3. Приоритеты и схема взаимодействия правил
├── AI_RULES.md              ← 4. Политика экономии токенов и ведения todo.md
├── SECURITY.md              ← 5. Матрица рисков Decision Policy (L0-L6)
├── SKILLS.md                ← 6. Индекс доступных навыков
└── todo.md                  ← 7. Текущий рабочий лог задач

Глобальный каталог навыков агента (в домашней директории пользователя)
└── skills/
    ├── windows/             ← Диагностика ОС и прикладного ПО
    ├── linux/               ← Логи, системные службы, Docker
    ├── git/                 ← Стандарты работы с Git и хуки
    └── session_management/  ← Сжатие контекста тяжелых сессий
```

### Преимущества архитектуры:
*   **Экономия токенов:** ИИ при запуске читает только `MANIFEST.md` и `AGENTS.md` (минимальный вес). Сложные инструкции и навыки подгружаются точечно, только когда они нужны.
*   **Четкие рамки (Decision Policy):** ИИ точно знает, какие команды выполнять автономно, а какие требуют согласования с разработчиком.

### Шаблон 1: [MANIFEST.md]

```markdown
L0-L6 (detailed in SECURITY.md)

UTF-8 BOM

Entry point:
AGENTS.md

Core rules:
AI_RULES.md

Decision policy:
SECURITY.md

Skills index:
SKILLS.md
```

### Шаблон 2: [ARCHITECTURE.md]
Описывает порядок применения файлов правил.
```markdown
# AI Rules & Security Architecture

Этот файл описывает архитектуру ИИ-контракта, логику взаимодействия файлов правил и приоритеты их применения.

## Приоритет правил при конфликтах
1. **SECURITY.md** — Ограничения безопасности (высший приоритет).
2. **AGENTS.md** — Базовый ИИ-контракт.
3. **AI_RULES.md** — Регламенты TOKEN-FIRST и ведения логов.
4. **Правила конкретных проектов** (внутри папок проектов).
5. **Глобальные навыки** (Skills).
```

### Шаблон 3: [AGENTS.md]
Точка входа для ИИ-агента.
```markdown
## Language Policy
- Всегда общайся со мной и выводи результаты только на русском языке.
- Комментарии к коду оставляй на английском или русском.

# Workspace AI Contract

Этот файл является точкой входа (контрактом) для ИИ-ассистента.

## 1. Системные правила и регламенты
Перед началом работы ассистент обязан прочесть эти документы для правильного принятия решений:
1. MANIFEST.md — Манифест проекта (читать в первую очередь).
2. ARCHITECTURE.md — Архитектура ИИ-правил и приоритеты.
3. AI_RULES.md — Общие правила, TOKEN-FIRST и ведение логов.
4. SECURITY.md — Матрица рисков Decision Policy (L0-L6) и подтверждения.
5. SKILLS.md — Индекс доступных навыков.

## 2. Локальный контекст
- Выполняемые задачи фиксируются в локальном логе: todo.md.
```

### Шаблон 4: [AI_RULES.md]
Политика экономии токенов (TOKEN-FIRST 4.0).
```markdown
# Общие правила работы и TOKEN-FIRST

Этот файл определяет правила оптимизации расхода токенов API.

## 1. TOKEN-FIRST Регламент
Перед выполнением любых операций чтения файлов или поиска ИИ обязан:
1.  **Оценить необходимость:** Если на вопрос можно ответить по контексту — чтение запрещено.
2.  **Анализ индекса:** Перед открытием любого файла >50 строк свериться с индексом проекта (`.ai/file_index.json`).
3.  **Порционное чтение:** Читать файлы частями (не более 200 строк за один раз через `view_file` с указанием строк).
4.  **Лимит сессии:** При достижении 10–12 шагов выдать предупреждение о необходимости сжатия сессии (запуск навыка `session_management`).

## 2. Фиксация прогресса и ошибок
*   **Рабочий лог (todo.md):** Все задачи фиксируются в [todo.md]. Невыполненные: `[ ]`, в процессе: `[/]`, завершенные: `[x]`.
*   **Автоисправление:** Разрешено максимум 3 попытки автоматического исправления ошибок сборки или тестов. На 4-ю — остановка и отчет разработчику.
```

### Шаблон 5: [SECURITY.md]
Уровни рисков и автономия.
```markdown
# Decision Policy и Матрица Рисков

Этот файл регулирует уровень автономии ИИ-ассистента.

## 1. Матрица уровней риска (L0 - L6)

| Уровень Риска | Описание | Что можно делать автономно | Требуется подтверждение |
| :--- | :--- | :--- | :--- |
| **L0: Чтение** | Чтение файлов, поиск текста, индексация. | Автономно (в пределах лимитов по строкам). | Не требуется. |
| **L1: Диагностика** | Запуск безопасных команд опроса (ping, get-process). | Разрешено для исследования проблемы. | |
| **L2: Изменение файлов** | Редактирование локального кода, создание скриптов. | Редактирование по согласованному плану. | Не требуется в рамках утвержденного плана. |
| **L3: Изменение инфры** | Изменение системных настроек ОС, реестра, сети. | Только исследование и планирование. | **Требуется подтверждение** перед изменением. |
| **L4: Сеть и Интернет** | Запросы к внешним API, Curl, скачивание пакетов. | Исследование доступности портов. | **Требуется подтверждение** на внешние запросы. |
| **L5: Git Push** | Отправка изменений в удаленный репозиторий. | Подготовка коммитов локально. | **Требуется подтверждение** (контролируется pre-push хуком). |
| **L6: Production** | Работа с продакшн-серверами, боевыми БД, службами. | Мониторинг логов и служб (read-only). | **Критический аппрув.** Любое изменение требует явного подтверждения. |
```

## 3. Настройка гибкой защиты Git Push (Вайтлист репозиториев)

Для предотвращения случайных пушей в неавторизованные репозитории на локальных машинах сотрудников настраивается глобальный pre-push хук.

### Шаг 1. Создание папки хуков
В домашней директории пользователя (например, `C:/Users/username/.git_hooks/`) создаются два файла:

#### Файл `allowed_remotes.json` (Вайтлист)
Сюда сотрудник вносит разрешенные маски репозиториев:
```json
{
  "allowed": [
    "github.com/your-organization/*",
    "github.com/your-username/*"
  ]
}
```

#### Файл `pre-push` (Скрипт хука)
Напишите исполняемый Bash-скрипт (заменив путь к вайтлисту на актуальный для конкретной машины):
```bash
#!/bin/sh

remote_name="$1"
remote_url="$2"
whitelist_file="C:/Users/username/.git_hooks/allowed_remotes.json" # Указать корректный путь

# 1. Проверяем наличие вайтлиста
if [ ! -f "$whitelist_file" ]; then
    echo "SECURITY ERROR: Whitelist file not found at $whitelist_file"
    exit 1
fi

# 2. Проверяем URL по вайтлисту через утилиты Git Bash (grep, sed)
matched=0
for pattern in $(grep -o '"[^"]*"' "$whitelist_file" | sed 's/"//g' | grep -v '^allowed$'); do
    regex=$(echo "$pattern" | sed 's/\*/.*/g')
    if echo "$remote_url" | grep -iq "$regex"; then
        matched=1
        break
    fi
done

if [ "$matched" = "1" ]; then
    exit 0
fi

# 3. Проверяем переменную принудительного обхода
if [ "$ALLOW_EXTERNAL_PUSH" = "1" ]; then
    echo "WARNING: Push to external repository allowed by ALLOW_EXTERNAL_PUSH=1."
    exit 0
fi

# 4. Блокируем пуш
echo ""
echo "========================================= SECURITY ALERT ========================================="
echo "PUSH BLOCKED: The remote repository URL is not in the whitelist:"
echo "  $remote_url"
echo ""
echo "To allow this push, add the pattern to: $whitelist_file"
echo "Or bypass it temporarily:"
echo "  In PowerShell: \$env:ALLOW_EXTERNAL_PUSH='1'; git push"
echo "  In CMD:        set ALLOW_EXTERNAL_PUSH=1 && git push"
echo "  In Git Bash:   ALLOW_EXTERNAL_PUSH=1 git push"
echo "=================================================================================================="
echo ""
exit 1
```

### Шаг 2. Активация хука в Git
Каждый сотрудник должен выполнить на своей локальной машине команду:
```bash
git config --global core.hooksPath C:/Users/username/.git_hooks
```
*(где указан путь к созданной директории с хуком)*
