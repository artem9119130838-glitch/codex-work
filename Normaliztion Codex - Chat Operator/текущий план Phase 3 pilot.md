# План: экономный Phase 3 pilot на существующей структуре marketing_db

## Summary

Не создаем новую тяжелую архитектуру.  
Используем уже существующие таблицы `email_messages`, `email_match_results`, `clients_intel`, `onec_owners`, `onec_contacts`.

Стратегия пилота:
1. берем **отдельную IMAP-папку** со старыми письмами;
2. загружаем **20–30 самых старых адресов/клиентов**;
3. сохраняем только **очищенные и полезные письма**, а не полный хвостатый архив;
4. матчим письма к `onec_owners/onec_contacts`;
5. сводку и тему держим в `clients_intel` и частично в `onec_owners`;
6. модель готовит только `draft`, без автоотправки.

## Key Changes

### 1. Что используем из текущей БД

#### `email_messages`
Оставляем ее как основное хранилище писем пилота.

Использование полей:
- `message_id` — дедуп
- `from_email`, `from_name`
- `subject`
- `received_at`
- `source_mailbox`
- `raw_payload` — только как сырой источник, не для постоянного чтения моделью
- `is_junk`, `junk_reason`
- `operator_comment`
- `thread_comment`
- `ai_summary`

Правило:
- модель **не читает raw_payload целиком**
- перед AI-анализом письмо очищается до короткой смысловой версии
- эту краткую версию храним либо:
  - в `ai_summary`, если хотим совсем без миграций,
  - либо в новом одном поле `body_clean_short text`, если решим добавить только одно полезное поле

Рекомендация:  
для пилота **лучше добавить ровно одно поле** в `email_messages`:
- `body_clean_short text`

Это дешевле, чем каждый раз заново чистить `raw_payload`, и не раздувает схему.

#### `email_match_results`
Оставляем как есть, без тяжелого расширения.

Она уже годится для:
- matched / needs_review
- confidence
- owner/contact linkage
- matched_email / matched_phone
- decision_reason

Нужная логика:
- одна запись результата на одно письмо
- при повторном прогоне обновляем решение, а не плодим мусор

#### `clients_intel`
Используем как **легкий клиентский агрегат**, а не создаем новую таблицу-профиль.

Туда складываем:
- `email`
- `source_id_1c`
- `company_name`
- `inn`
- `ai_priority`
- `ai_summary`
- `last_topic`
- `current_status`
- `next_followup_at`
- `updated_at`

Правило:
- `clients_intel.ai_summary` = короткая накопительная сводка клиента
- `clients_intel.last_topic` = текущая тема / последняя релевантная тема
- `clients_intel.current_status` = например `pilot_matched`, `needs_review`, `draft_ready`, `stale_lead`
- `clients_intel.next_followup_at` = дата следующего ручного касания/напоминания

#### `onec_owners`
Используем как источник клиентских фактов и место для ограниченного enrichment:
- `owner_topic`
- `ai_client_segment`
- `ai_email_summary`
- `sent_articles_history`
- `last_contact_date`

Правило:
- `onec_owners` не превращаем в архив писем
- туда пишем только **сжатые выводы**, а не поток сообщений

### 2. Что не делаем

На пилоте **не делаем**:
- новую большую таблицу истории переписки
- отдельную таблицу draft-кампаний
- хранение полного чистого архива всех тредов
- анализ всего почтового ящика
- автоотправку
- автоответ на входящие
- автоматическое создание задач в Bitrix

### 3. Как экономно обрабатывать письма

#### Источник
Используем **одну IMAP-папку пилота**.

Да, в n8n можно указывать конкретную папку/mailbox.  
Это лучший вариант:
- не трогаем весь ящик,
- ограничиваем выборку,
- не плодим дубли.

#### Выборка пилота
Берем:
- 20–30 самых старых адресов/клиентов
- период: сначала `2022`, если мало — ранний `2023`
- лучше вручную отобранную пилотную папку

#### Очистка письма
Перед записью/анализом:
1. дедуп по `message_id`
2. чистка `subject`
3. удаление quoted tail:
   - `Re:`
   - `FW:`
   - `-----Original Message-----`
   - `On ... wrote:`
   - строки `>`
4. выделение только **новой смысловой части**
5. ограничение по длине

Результат:
- в `raw_payload` остается сырье,
- в `body_clean_short` или `ai_summary` остается полезный срез.

### 4. Matching-алгоритм первой итерации

Порядок:
1. точный match по `from_email` в `onec_contacts.emails`
2. точный match по `from_email` в `onec_owners.owner_all_emails` / `owner_search_emails`
3. match по телефону, если письмо от формы и телефон извлечен
4. match по `company_name + contact_name`
5. если не нашли:
   - `decision = needs_review`
   - `matched_level = unmatched`

Для `site_form` писем:
- `from_email` формы не считаем клиентским email
- ищем email/телефон/ИНН/компанию внутри тела формы
- только по ним пытаемся привязать к клиенту

### 5. Как будет работать модель

Модель не читает всю переписку.  
Ей дается **компактный пакет**:

- owner/contact данные из `onec_owners` / `onec_contacts`
- 3–7 очищенных писем клиента:
  - первое найденное письмо
  - 1–3 ранних письма
  - последнее релевантное письмо
- правила из curated KB:
  - `70_input_client_context.md`
  - `80_template_selection_rules.md`
  - `90_pre_send_checklist.md`
  - `95_article_selection_rules.md`

#### Алгоритм модели
Шаг 1. Определить:
- кто клиент
- как начался диалог
- какая была первая тема
- по какому бренду / направлению / задаче он пришел

Шаг 2. Сжать это в:
- `ai_summary`
- `last_topic`
- `ai_client_segment` или `ai_priority`, если это уже уместно

Шаг 3. Подготовить:
- тему мягкого касания
- 1 подходящую статью
- 1 черновик письма

#### Куда пишем результат
- краткую сводку клиента — в `clients_intel.ai_summary`
- последнюю тему — в `clients_intel.last_topic`
- при необходимости сжатую email-oriented сводку — в `onec_owners.ai_email_summary`
- историю отправленных статей позже — в `onec_owners.sent_articles_history`

На пилоте сам draft можно:
- либо хранить временно в `operator_comment` / `thread_comment`,
- либо сделать **одну отдельную markdown/JSON выгрузку по пилоту**, чтобы не менять БД лишний раз.

Рекомендация для пилота:  
**не добавлять draft-поля в БД сразу**, а сначала делать draft-выгрузку в проектный файл/табличный отчет. Это самый экономный старт.

### 6. Структура чатов

#### Главный чат
Только:
- стратегия
- контроль канона
- прием результатов

#### Рабочий чат 1: `Phase 3 mail intake + matching`
Делает только:
- IMAP-папка пилота
- intake писем
- dedup
- clean body
- запись в `email_messages`
- запись в `email_match_results`

Deliverable:
- 20–30 клиентов/адресов загружены и размечены

#### Рабочий чат 2: `Phase 3 AI summary + drafts`
Делает только:
- берет matched клиентов пилота
- обновляет `clients_intel`
- при необходимости пишет краткий enrichment в `onec_owners`
- делает draft preview
- без send

Deliverable:
- 5–10 хороших черновиков и понятные сводки

Правило:
- один чат = один узкий deliverable
- после результата чат закрываем/замораживаем
- не тащим один чат на сотни тысяч токенов

## Test Plan

### Этап 1. Intake + matching
Успех:
- n8n читает письма только из пилотной папки
- в `email_messages` нет дублей по `message_id`
- хвосты переписки не тащатся в AI-контекст
- `email_match_results` заполняется
- для не найденных есть `needs_review`

### Этап 2. Client summary
Успех:
- для 20–30 клиентов появляются краткие сводки в `clients_intel`
- определена первая тема и последняя тема
- обновлены `current_status` и `updated_at`

### Этап 3. Draft preview
Успех:
- для 5–10 старых клиентов получены понятные draft-письма
- статьи подбираются осмысленно
- ничего не отправляется автоматически

## Assumptions

- База `marketing_db` уже содержит достаточно owner/contact данных для пилотного матчинга.
- Мы не перестраиваем схему заново, а используем текущие таблицы как основу.
- Для пилота допустима максимум одна легкая миграция в `email_messages` (`body_clean_short text`), если без нее слишком неудобно.
- Draft-ы на первом этапе не обязаны храниться в БД как отдельная сущность; можно выдавать их в проектный артефакт или временный отчет.
- Автоотправка — не часть текущего этапа.
