# Decision Policy и Матрица Рисков (Личный контур)

Этот файл регулирует уровень автономии ИИ-ассистента и определяет правила запуска команд и редактирования файлов.

## 1. Матрица уровней риска (L0 - L6)

Каждая задача перед выполнением классифицируется ИИ по уровню риска.

| Уровень Риска | Описание | Что можно делать автономно | Требуется подтверждение пользователя |
| :--- | :--- | :--- | :--- |
| **L0: Чтение** | Чтение файлов, поиск текста, индексация. | Полностью автономно (в пределах лимитов по строкам). | Не требуется. |
| **L1: Диагностика** | Запуск безопасных команд опроса (ping, get-process, проверка существования файлов). | Разрешено запускать для исследования проблемы. | Не требуется. |
| **L2: Изменение файлов** | Редактирование локального кода, создание новых скриптов. | Редактирование по согласованному плану или по прямому указанию. | Не требуется в рамках утвержденного плана. |
| **L3: Изменение инфры** | Изменение системных настроек Windows, реестра, сетевых адаптеров. | Только исследование и планирование. | **Требуется подтверждение** перед применением изменений. |
| **L4: Сеть и Интернет** | Запросы к сторонним API, скачивание пакетов, Curl. | Исследование доступности портов. | **Требуется подтверждение** на внешние запросы. |
| **L5: Git Push** | Отправка изменений в удаленный репозиторий. | Подготовка коммитов локально. | **Требуется подтверждение** (контролируется pre-push хуком). |
| **L6: Production** | Работа с сервером VPS, БД 1С/PostgreSQL, перезапуск служб. | Мониторинг логов и служб (read-only). | **Критический аппрув.** Любое изменение инфры VPS (apache reload, docker-compose, перезапуск 1С) требует явного "Да". |

## 2. Правила минимизации UAC и предотвращения зацикливания

1.  **Исследование без лишних вопросов:** Диагностические и поисковые команды (L0-L1) ИИ обязан запускать самостоятельно, чтобы не отвлекать пользователя.
2.  **Группировка команд:** Если требуется запустить несколько команд L2-L6, ИИ обязан объединить их в одну строку (например, через `&&` или `;` в PowerShell/Bash), чтобы пользователь подтверждал их за один клик.
3.  **Использование Whitelist для Git:** Пуши разрешены только в репозитории из вайтлиста `C:/codex_home/.git_hooks/allowed_remotes.txt` (проверяется локально pre-push хуком).
4.  **Правило "Stop-on-Fail" для L3-L6:** Если команда с повышенными привилегиями (UAC, sudo, перезапуск служб) или запрос доступа завершились ошибкой, ИИ **запрещено** повторно просить разрешения на аналогичные команды "наугад" (например, с другими флагами или в цикле). ИИ должен остановиться после **первой же неудачи**, подробно доложить об ошибке и передать управление пользователю.
5.  **Запрет "слепых" системных действий (1С на VPS):** Категорически запрещено пытаться устанавливать, настраивать или запускать серверные/клиентские службы 1С:Предприятия на VPS «вслепую» (без наличия проверенного Runbook в `codex_kb/30_runbooks/` или прямых пошаговых инструкций от пользователя).
6.  **Объединение команд в скрипт (Script-First):** Если для решения задачи (например, деплоя, настройки окружения, устранения системных ошибок) требуется выполнить последовательность из более чем 3-х команд в терминале (локально или по SSH на VPS), ИИ обязан объединить эти команды в один исполняемый скрипт (например, `.sh` для Linux/VPS или `.ps1` для Windows), сохранить его в проекте или временной папке и запросить разрешение на выполнение **одного этого скрипта**, вместо пошагового выполнения команд с отдельными запросами.
